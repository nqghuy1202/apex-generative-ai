**Milestone — latency root cause VERIFIED via packet capture (2026-06-30 round 2):** the CRM agent's ~50–130s latency was traced end-to-end. Ruled out (each disproven by data): dual-socket NUMA (`numactl --hardware` shows **1 NUMA node** — BIOS interleave; server B is 2× Xeon E5-2680 v2, 16 cores, no AVX2), slow generation (actually ~12 tok/s, fine), and an injected timestamp (no date in the prompt). **Actual root cause (tcpdump of APEX→Ollama):** APEX runs **2 LLM calls per question** (pick-tool, then read-result+answer); on the 2nd call APEX prepends a security system message starting with a **random `UNTRUSTED-DATA-<hex>` marker** (anti-prompt-injection, regenerated **every request** — confirmed 4 distinct markers in one chat session). Sitting at token ~1, it makes Ollama's KV **prefix-cache miss every call** (`sim=0.001`, `cached n_tokens≈0`) → full re-prefill of ~1700–2900 tokens @ ~50 tok/s. **No documented switch to disable it.** Only lever left = shrink the prefill token count. Authored compressed config **§0-OPT** in `crm_leads_agent_prompts.md` (system prompt ~700 tok + nested Description/Data Description) and report `technical-crm-leads-latency-numa-round2-research-2026-06-30.md`. Measured baseline: "có bao nhiêu khách hàng tiềm năng?" = **~1m7s**. Realistic ceiling on this 2013 CPU: tool-using questions ~30–45s (not <10s). Open TODO: (1) re-measure after §0-OPT; (2) fix `query_lead_metrics` error when `p_group_by=owner`. Memories: `apex-untrusted-data-marker-cache-killer`, `llm-cpu-prefill-cache-bottleneck`, `apex-ai-skip-call2-hardexit`.

**Milestone — latency round 3: cut the 2nd LLM call (2026-07-01):** re-measured baseline **after §0-OPT is still ~1m7s** for a tool question → prompt-compression is exhausted. Ran `bmad-technical-research` (report `technical-crm-leads-latency-round3-skip-call2-research-2026-07-01.md`). **Decisive new finding:** APEX 26.1 `APEX_AI.GENERATE`/`APEX_AI.CHAT` support **inline PL/SQL tools** (`apex_ai.t_tools`) plus a **response-handler "hard exit" that stops the agentic loop BEFORE tool results are sent back to the LLM**. Rebuilding the agent as a PL/SQL `APEX_AI.GENERATE` page = **1 LLM call** (LLM picks tool+args → tool SQL runs → PL/SQL formats the result into a Vietnamese answer → hard-exit) instead of the current 2-call flow. This **deletes the ~35–71s uncacheable call-2 re-prefill** → projected ~1m7s → 15–25s cold, 3–6s warm → **hits <30s**. Secondary lever (untried, ~free, 37GB RAM spare): `num_batch` 512→1024/2048 in Modelfile = ~20–40% faster prefill. **Ruled out:** dropping to 1.5b (tool-calling collapses: 3B 71% vs 1.5B 27%), hardware upgrade (out of scope). **Critical:** tuning Ollama alone while keeping the 2-call architecture still ceilings at ~45s — the hard-exit rewrite is mandatory to reach <30s. **Next session:** (1) add `num_batch 1024` to `Modelfile.qwen3-erp`, measure prefill tok/s; (2) build a PoC APEX page calling `APEX_AI.GENERATE` with one inline tool (`query_lead_metrics`) + hard-exit, confirm only 1 LLM call via tcpdump. **Key project conventions (see memory):**
- PK columns use explicit **SEQUENCE + `.NEXTVAL`**, never `GENERATED AS IDENTITY` (system rule — `use-sequences-not-identity`).
- **Accent-strip / Vietnamese normalization — two functions, know which:** the **CRM_LEADS** files (`crm_leads_agent_tools.sql`, `crm_leads_agent_generate_poc.sql`) now use **`bodau`** (pure-SQL `UPPER(TRANSLATE(...))` in `sql/bodau.sql`, added 2026-07-02). The older **`customers`** files still use **`mle_norm`** (MLE/JS in `sql/mle_text_normalize.sql`). They are NOT interchangeable in one comparison — `bodau` returns UPPERCASE, `mle_norm` lowercase — use the SAME function on both sides. **Why `bodau` for scale:** it is pure SQL + `DETERMINISTIC`, so a **function-based index `bodau(col)` IS usable** (unlike per-row MLE/JS `mle_norm`), so wrapping `bodau(col)` in `WHERE` no longer forces a full scan *if* the FBI exists AND the predicate is `bodau(col)=bodau(:p)` or `bodau(col) LIKE 'X%'` (leading `%` still scans). Create FBIs on `cle_name`/`customer`/`source` before go-live (commented in `bodau.sql`). `TRANSLATE` maps char-by-char so the `from`/`to` strings must be equal length (134=134) with no stray whitespace. `mle_norm` remains valid for the small `customers` dataset (memory `mle-normalize-at-write-not-query`).
- The earlier `HTTP 400 invalid input` was only the **Test Connection** chat probe failing against embedding-only bge-m3 — it does NOT block real embedding use. Provider = Ollama, Base URL = host:port only (no `/v1/embeddings`).
- Operational gotcha: run steps in order (seq → insert documents → chunk → UPDATE embedding + COMMIT) before querying; a NULL `embedding` makes `VECTOR_DISTANCE` return NULL.
- **CPU LLM speed:** the dominant cost was **thinking-mode** generating runaway output. On Qwen3 you CANNOT disable it (`/no_think` and `"think": false` both ignored) — for ERP tool-calling use a **pure-instruct model: `qwen2.5:3b-instruct`** (escalate to `qwen2.5:7b-instruct` if 3b picks the wrong tool). Always write the system prompt in **Vietnamese with diacritics**. `ORA-20960` from the AI Assistant means the model emitted an invalid tool-call (weak/embedding-only models trigger this) — debug by tailing `journalctl -u ollama -f` while reproducing. The model lives on **server B (Linux/bash)**, not this Windows box — don't paste PowerShell into its bash shell.
- **APEX AI write/action tools:** the PL/SQL tool body is compiled as a **function that MUST `RETURN <varchar2>`** (bare `RETURN;` → PLS-00503; `apex_ai.set_tool_result` does not apply to this tool type). APEX limits tool code to **4000 chars** → wrap logic in a package and call it in one line. **Never** call `apex_ai.get_vector_embeddings` inline inside a chat tool — loading bge-m3 mid-conversation evicts qwen3-erp on the shared CPU and the 2nd agentic turn exceeds APEX's 180s HTTP timeout (`ORA-29276`); defer embeddings to the `CRM_LEADS_EMBED_JOB` background job. Declare exactly the input parameters the model sends (mismatch → `ORA-20960`); do not declare OUT params. Enable **Requires Confirmation** on write tools.
- **CPU latency is dominated by prompt prefill**, not generation. Biggest lever is the **KV prefix cache**: keep the system-prompt + tool-schema prefix byte-identical across turns (no timestamps/user/session at the top) so a repeated query reuses it (`journalctl` shows `cached n_tokens` ≈ `task.n_tokens` on a hit; `sim=0.001` means a miss). Set `num_thread`=physical cores, `OLLAMA_NUM_PARALLEL=1`, `OLLAMA_MAX_LOADED_MODELS=2`, `OLLAMA_KEEP_ALIVE=-1`. Tool-field meanings (Description=when, Data Description=column meaning, Execution Point Augment-vs-OnDemand) in memory `apex-ai-tool-fields`.
- **BUT prefix-cache cannot help the tool-result (2nd) LLM call** because APEX prepends a random `UNTRUSTED-DATA-<hex>` security marker that changes every request → guaranteed cache miss → full re-prefill. So for tool-using questions the only real speedup is **fewer prefill tokens** (compressed system prompt + tool schemas = §0-OPT; smaller tool results; `num_predict` low). `<10s` is not achievable for tool questions on the AVX2-less E5-2680 v2. Server B is **1 NUMA node** (BIOS interleave) → `numactl --membind` is moot, use `num_thread 16`. Keep `num_ctx 4096` (agentic turns reach ~2300 tokens; 2048 truncates). Diagnose payloads with `tcpdump -w file.pcap 'tcp port 11434 and src host <apex-ip>'` then `strings file.pcap | grep ...`. Memory: `apex-untrusted-data-marker-cache-killer`.

Full reports in `_bmad-output/planning-artifacts/research/`. Persistent memory: `apex-ollama-bge-m3-embedding`, `apex-vector-rag-table-design`, `use-sequences-not-identity`, `erp-local-llm-model-choice`, `llm-cpu-speed-optimization`, `apex-ai-write-tool-pattern`, `apex-ai-tool-fields`, `server-b-ollama-infra`, `crm-leads-ranking-tool-gap`, `mle-normalize-at-write-not-query`, `apex-untrusted-data-marker-cache-killer`, `llm-cpu-prefill-cache-bottleneck`, `apex-ai-skip-call2-hardexit`, `apex-google-sheets-sync`.
