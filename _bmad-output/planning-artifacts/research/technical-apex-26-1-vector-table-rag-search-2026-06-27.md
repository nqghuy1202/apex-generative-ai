# Technical Research — Tạo bảng VECTOR + RAG vector search trên APEX 26.1 / DB 26ai

**Date:** 2026-06-27 · **Researcher:** Claude (bmad-technical-research) · **For:** Gia Huy
**Scope:** Schema 2 bảng (documents ↔ doc_chunks), chunking tài liệu dài tiếng Việt, sinh embedding bằng `apex_ai.get_vector_embeddings` ('apex-embed' → bge-m3:latest, dim 1024, COSINE), index HNSW/IVF, APPROX similarity search.

---

## Phần A — DDL (schema 2 bảng cho RAG)

```sql
-- Bảng cha: mỗi tài liệu 1 dòng
CREATE TABLE documents (
  doc_id     NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  title      VARCHAR2(500),
  source     VARCHAR2(1000),         -- tên file / URL gốc
  doc_text   CLOB,                   -- toàn văn tài liệu dài
  created_at TIMESTAMP DEFAULT SYSTIMESTAMP
);

-- Bảng con: mỗi chunk 1 dòng
CREATE TABLE doc_chunks (
  chunk_id   NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  doc_id     NUMBER NOT NULL REFERENCES documents(doc_id),
  chunk_seq  NUMBER,                 -- thứ tự chunk trong tài liệu
  chunk_text VARCHAR2(4000),         -- nội dung chunk (≤4000 ký tự)
  embedding  VECTOR(1024, FLOAT32)   -- KHỚP bge-m3 (1024 chiều)
);
```

> **Quyết định quan trọng — dimension:** Tài liệu CREATE VECTOR INDEX nói "chỉ tạo index trên cột VECTOR mà mọi vector cùng số chiều". Để an toàn cho index HNSW/IVF, **khai báo `VECTOR(1024, FLOAT32)`** thay vì `VECTOR` thuần như video. bge-m3 = 1024 chiều. [nguồn: CREATE VECTOR INDEX 26 — "create a vector index only on a VECTOR column containing vectors that all have the same number of dimensions"]

## Phần B — Chunking + nạp embedding (PL/SQL)

Dùng `DBMS_VECTOR_CHAIN.UTL_TO_CHUNKS` (DB-native chunking) để cắt `doc_text` → nhiều dòng `doc_chunks`, sau đó loop gọi `apex_ai.get_vector_embeddings` cho từng chunk.

```sql
BEGIN
  -- B1: cắt mỗi tài liệu thành chunks, đổ vào doc_chunks
  FOR d IN (SELECT doc_id, doc_text FROM documents) LOOP
    INSERT INTO doc_chunks (doc_id, chunk_seq, chunk_text)
    SELECT d.doc_id,
           JSON_VALUE(c.column_value, '$.chunk_id'   RETURNING NUMBER),
           JSON_VALUE(c.column_value, '$.chunk_data')
    FROM   dbms_vector_chain.utl_to_chunks(
             d.doc_text,
             JSON('{ "by":"characters", "max":"3000", "overlap":"300",
                     "split":"recursively", "normalize":"all" }')
           ) c;
  END LOOP;

  -- B2: sinh embedding cho từng chunk
  FOR ch IN (SELECT chunk_id, chunk_text FROM doc_chunks WHERE embedding IS NULL) LOOP
    UPDATE doc_chunks
       SET embedding = apex_ai.get_vector_embeddings(
                         p_value             => ch.chunk_text,
                         p_service_static_id => 'apex-embed')
     WHERE chunk_id = ch.chunk_id;
  END LOOP;

  COMMIT;
END;
/
```

**Tham số chunking** [nguồn: UTL_TO_CHUNKS docs]: `by` = characters|words|vocabulary (default words); `max` chars 50–4000 (default 100); `overlap` 5%–20% của max; `split` default `recursively`; `normalize` default `all`. Input nhận cả CLOB và VARCHAR2.

> ⚠️ **Tiếng Việt:** tham số `language` của UTL_TO_CHUNKS dựa trên NLS, **tài liệu KHÔNG liệt kê tiếng Việt rõ ràng** [chưa xác minh]. → Khuyến nghị dùng **`by:"characters"`** (không phụ thuộc ngôn ngữ) + `normalize:"all"`, bỏ `language`, để tránh lỗi tokenization tiếng Việt. `max:"3000"` chừa biên dưới 4000 cho an toàn cột VARCHAR2(4000).

> ⚠️ **Ép kiểu:** ở session trước `apex_ai.get_vector_embeddings` trả VECTOR dùng trực tiếp được. Nếu môi trường trả CLOB/JSON, bọc `TO_VECTOR(...)`. [chưa xác minh kiểu trả về tuyệt đối]

## Phần C — Vector Index (chọn HNSW hoặc IVF)

**HNSW** (graph, in-memory — truy vấn nhanh, chính xác cao; cần vector pool bộ nhớ):
```sql
CREATE VECTOR INDEX doc_chunks_hnsw_idx ON doc_chunks (embedding)
ORGANIZATION INMEMORY NEIGHBOR GRAPH
DISTANCE COSINE
WITH TARGET ACCURACY 95
PARAMETERS (type HNSW, neighbors 40, efconstruction 500);
```

**IVF** (partition-based — tốn ít RAM hơn, hợp data rất lớn):
```sql
CREATE VECTOR INDEX doc_chunks_ivf_idx ON doc_chunks (embedding)
ORGANIZATION NEIGHBOR PARTITIONS
DISTANCE COSINE
WITH TARGET ACCURACY 95
PARAMETERS (type IVF, neighbor partitions 10);
```

[nguồn: CREATE VECTOR INDEX 26]. Tham số HNSW: `neighbors` (M) 2–2048, `efconstruction` >0–65535. IVF: `neighbor partitions` 1–10⁷. Default distance = COSINE.

**Khi nào dùng cái nào:** HNSW = mặc định cho độ chính xác + tốc độ khi data vừa phải và đủ RAM (cần cấu hình `vector_memory_size` cho vector pool [chưa xác minh chi tiết init param]). IVF = khi tập vector rất lớn / hạn chế bộ nhớ. Lúc TEST data nhỏ → HNSW là đủ và đơn giản.

## Phần D — APPROXIMATE similarity search

```sql
DECLARE
  l_qvec   VECTOR;
  l_answer VARCHAR2(4000);
BEGIN
  l_qvec := apex_ai.get_vector_embeddings(
              p_value             => 'Câu hỏi của người dùng bằng tiếng Việt',
              p_service_static_id => 'apex-embed');

  FOR r IN (
    SELECT d.title, dc.chunk_text,
           VECTOR_DISTANCE(dc.embedding, l_qvec, COSINE) AS dist
    FROM   doc_chunks dc
    JOIN   documents  d ON d.doc_id = dc.doc_id
    ORDER  BY dist
    FETCH  APPROX FIRST 5 ROWS ONLY      -- APPROX = dùng vector index
  ) LOOP
    dbms_output.put_line(r.dist || ' | ' || r.title || ' | ' ||
                         SUBSTR(r.chunk_text,1,80));
  END LOOP;
END;
/
```

- `FETCH APPROX FIRST n ROWS ONLY` báo optimizer dùng vector index (approximate). Bỏ `APPROX` → exact full-scan. [nguồn: Vector search query docs 26 — APPROX(IMATE) keyword]
- JOIN ngược `documents` để biết chunk thuộc tài liệu nào (lợi thế schema 2 bảng cho RAG).

## Phần E — Best practice & lỗi thường gặp

- **Index chỉ "thắng" exact khi data đủ lớn.** Vài chục–vài trăm dòng test, exact scan đã rất nhanh; APPROX có thể không khác biệt. Đừng kết luận index hỏng khi data nhỏ.
- **Dimension mismatch** = lỗi phổ biến nhất: cột phải `VECTOR(1024,FLOAT32)` khớp output bge-m3, nếu không CREATE INDEX/INSERT sẽ lỗi.
- **APPROX rơi về exact**: nếu index chưa tạo/không hợp lệ, query vẫn chạy nhưng là exact. Kiểm bằng `EXPLAIN PLAN` xem có dùng vector index không.
- **Target accuracy**: 90–95 là cân bằng tốt; cao hơn → chậm/tốn RAM.
- **Chunk overlap** 10–20% giúp giữ ngữ cảnh giữa các chunk cho RAG.
- **Commit** sau khi nạp; cân nhắc batch nếu tài liệu rất nhiều (mỗi chunk = 1 call REST tới Ollama → chậm, có thể giới hạn tốc độ).
